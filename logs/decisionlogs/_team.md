# Decisionlog · 팀 공용 (CareerPath AX for Counselors)

팀 전체 방향 결정을 기록한다. 대상: 문제 선정, 기술 스택 선택, 구현 범위 확정·축소, 데이터 처리 방식, 보안·개인정보 처리, 테스트·검증 방식, 배포·제출·공유 방식, 큰 구조 변경. (규칙: CLAUDE.md §11.3)

> 이 파일의 팀 결정은 **D-001부터 독립 번호**를 사용한다. PRD(`docs/prd/careerpath-ax-planning.md`) §14의 결정 기록 D-001~D-013은 팀 공용 결정이 아니라 **김윤서·박하린의 주제 확정 브레인스토밍 판단 기록**으로, 각자의 개인 결정 로그(`김윤서.md` D-KYS-001~013, `박하린.md` D-PHR-001~013)로 이관되었다. PRD의 결정을 참조할 때는 혼동을 피하기 위해 `PRD D-00N` 또는 `D-KYS-00N`/`D-PHR-00N`으로 표기한다.

```
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### D-008 · 포트폴리오 분석·NCS 능력단위는 MVP 제외, 확장 기능으로 표기
**상황**
- 3일차 회의에서 기획서 범위를 정하며 포트폴리오 분석·NCS 능력단위 매칭을 넣을지 논의.

**검토한 선택지**
1. MVP에 포함
2. MVP 제외, "확장 가능성"으로만 표기

**결정**
- 2안. 하루 구현 범위를 넘어서므로 MVP에서 제외하고 기획서에는 확장 기능으로 완곡히 표기.

**근거**
- 구현 부담이 크고, 억지로 넣으면 핵심(질문 리스트·공고 점수화·트렌드)의 완성도를 해친다.

**영향**
- 기획서 범위 표현을 "확장 가능성"으로 조정. PRD §10(MVP 제외 목록)과 정합 확인 필요.

**재검토 조건**
- 본선 진출 후 구현 여력이 생기거나 심사 피드백으로 요구되는 경우.

### D-007 · 팀명 GPS(Great-Path-Seeker) 확정
**상황**
- 팀명 미정. 후보로 GPS, 커리어 파일럿, 네비 AX 등이 나옴.

**검토한 선택지**
1. GPS(Great-Path-Seeker)
2. 커리어 파일럿
3. 네비 AX 등

**결정**
- 투표로 **GPS(Great-Path-Seeker)** 확정.

**근거**
- 발표 시 부담이 적고 의미(좋은 경로 탐색)가 서비스와 맞다는 다수 의견.

**영향**
- 원격 저장소명은 `Great-Path-Seekers`(복수형)로 이미 생성됨 — 표기 통일 여부 `확인 필요`.

**재검토 조건**
- 상표·중복 문제 등으로 팀명 변경이 필요한 경우.

### D-006 · 점수화는 TF-IDF 초안 + Cosine 유사도, 부족 시 AI 에이전트로 고도화
**상황**
- 공고 적합도 점수화 방식 논의. TF-IDF가 기존 서비스 대비 단순·차별화 부족이라는 우려 제기.

**검토한 선택지**
1. 처음부터 임베딩 등 고도화 기법
2. TF-IDF + Cosine 유사도로 초안, 부족하면 AI 에이전트 연계로 고도화

**결정**
- 2안. MVP는 TF-IDF 기반 점수화 + Cosine 유사도로 구현하고, 부족하면 AI 에이전트를 엮어 고도화.

**근거**
- 하루 구현 범위에서 빠르고 설명 가능. 차별화는 상담사 의견 반영·상담 이력 비교 등으로 보강.

**영향**
- 기획서에 "TF-IDF 초안 후 고도화 예정"으로 표기. PRD D-008(TF-IDF 채택)과 정합.

**재검토 조건**
- TF-IDF로 차별화가 부족하다고 판단되거나 데이터가 충분해 임베딩이 유효할 때.

### D-005 · 목표 직무를 데이터 분석으로 한정 (타 직무는 확장)
**상황**
- 대상 직무 범위 논의. 여러 직무를 넓게 다룰지, 하나로 좁힐지.

**검토한 선택지**
1. 여러 직무(PM·마케터 등) 동시
2. 데이터 분석 직무 하나로 한정, 이후 확장

**결정**
- 2안. 해커톤(하루 구현) 범위를 고려해 **데이터 분석 직무로 한정**하고 타 직무는 확장으로 둔다.

**근거**
- 하루 안에 완성하려면 범위를 좁혀 집중해야 한다. 직무 내 자주 쓰는 역량은 크게 안 변한다는 판단.

**영향**
- 샘플 데이터·질문 리스트·공고 키워드를 데이터 분석 직무 중심으로 설계.

**재검토 조건**
- 본선 확장 시 PM·마케터 등 추가 직무 지원.

### D-004 · 핵심 구현 기능 3가지 확정
**상황**
- 하루 구현 가능한 MVP 핵심 기능을 추려야 함(초기 기획엔 8~9개 기능이 있었음).

**검토한 선택지**
1. 기획서의 다수 기능을 폭넓게 구현
2. 상담 준비에 핵심적인 3가지로 압축

**결정**
- 2안. 핵심 기능을 **① 질문 리스트 생성 ② TF-IDF 공고 점수화·추천 ③ 3개월 트렌드 비교** 3가지로 확정.

**근거**
- 상담사가 상담 전에 가장 필요로 하는 것(구직자 실제 수준 파악·적합 공고·최신 트렌드)에 집중. 하루 내 구현 가능.

**영향**
- 이후 구현·기획서·데모 시나리오는 이 3기능 중심. 상담사 의견 반영·상담 이력 비교는 부가.

**재검토 조건**
- 데이터 소스 확보 실패 등으로 특정 기능 구현이 불가할 때.

### D-003 · 회의록은 구조화 요약본만 git 관리(원문은 Notion 전용), meeting-notes 스킬로 표준화
**상황**
- 회의록을 Notion 녹음 녹취록 기반으로 관리하기로 함. 매 회의 반복 작업을 스킬로 표준화하고 싶다. 초기엔 받아쓰기 전문 + 구조화 요약 2파일을 검토했으나, 3일차 회의록 실제 생성 시 녹취록 원문에 개인정보(팀원 인턴 제안 상세)·사담이 다수 포함됨을 확인.

**검토한 선택지**
1. 받아쓰기 전문 + 구조화 요약 2파일을 모두 git 저장 (원문에 개인정보 노출 위험, 히스토리 재작성으로도 완전 제거 어려움)
2. 원문은 접근 통제되는 팀 Notion에만 두고, git에는 구조화 요약본 1파일만 저장

**결정**
- 2안 채택. git에는 `logs/meetinglogs/YYYY-MM-DD_<회의명>.md`(구조화 요약본) 1파일만 둔다. 녹취록 원문(받아쓰기 전문)은 저장하지 않고 팀 Notion에서만 보관·공유한다. Notion 녹취록은 `.claude/skills/meeting-notes/` 스킬로 읽어 요약을 생성한다.

**근거**
- 한 번 원격(git)에 올라간 개인정보는 캐시·클론·포크로 남아 히스토리 재작성으로도 완전히 되돌리기 어렵다. 원문은 권한 통제되는 Notion에 두는 편이 안전하다. 요약본만으로 회의 내용 파악·의사결정 추적에는 충분하다.

**영향**
- D-002의 단일 파일 형식(`YYYY-MM-DD_N차.md`)과, 초기 검토했던 폴더+받아쓰기.md 구조는 이 방식으로 대체됨. CLAUDE.md·AGENTS.md §11.6, meetinglogs/README.md, SKILL.md, summary-guide.md 개정.
- 요약본에도 개인 신상·급여·연락처·사담은 옮기지 않는다. 요약은 팀 결정 후보·액션 아이템을 표시만 하고, 실제 `_team.md` D-ID 등록·worklog 반영은 사람이 한다(스킬은 발견 사항 보고).
- 스킬은 `NOTION_회의록_DB_URL`(회의록 DB)로 검색하고 `notion-fetch(include_transcript:true)`로 원문을 읽는다. Notion 커넥터 인증 필요.

**재검토 조건**
- Notion 녹음/녹취록 기능을 더 이상 쓰지 않거나, 원문을 git에 보존해야 할 필요(개인정보 없는 회의 등)가 생기는 경우.

### D-002 · 회의록 폴더 logs/meetinglogs/ 신설
**상황**
- 팀 회의(1~3차)가 진행 중이나 회의록을 기록할 표준 위치와 형식이 없었다. 김재천이 logs/ 하위 회의록 폴더 생성을 요청했다.

**검토한 선택지**
1. `meeting minutes` — 공백 포함, 기존 명명 패턴과 불일치
2. `meetinglogs` — 기존 `worklogs`·`decisionlogs`와 동일한 패턴(소문자·복수형·공백 없음)

**결정**
- `logs/meetinglogs/`로 생성. 파일 이름은 `YYYY-MM-DD_N차.md`, 회의 1건당 파일 1개.

**근거**
- 기존 폴더 명명 일관성 유지, 경로에 공백이 없어 도구·스크립트 취급이 안전하다.

**영향**
- CLAUDE.md·AGENTS.md에 §11.6(회의록 규칙) 추가, README 구조 갱신.
- 회의 중 팀 결정은 회의록에만 남기지 않고 `_team.md`에 D-ID로 기록 후 회의록에서 참조한다.

**재검토 조건**
- 회의록 양이 많아져 월별 하위 폴더 등 세분화가 필요해지는 경우.

### D-001 · 운영 로그를 logs/ 폴더 하위로 통합

**상황**

- 셋업 직후 루트에 `worklogs/`, `decisionlogs/`, `Troubleshootinglog.md`가 흩어져 있어 루트가 복잡했다. 김재천이 로그 관련 항목을 하나의 상위 폴더로 묶도록 요청했다.

**검토한 선택지**

1. 현행 유지(루트에 분산)
2. `logs/` 상위 폴더를 만들어 세 항목을 하위로 이동

**결정**

- `logs/` 폴더를 생성하고 `logs/worklogs/`, `logs/decisionlogs/`, `logs/Troubleshootinglog.md`로 이동한다.

**근거**

- 운영 로그가 한 폴더에 모여 루트가 단순해지고, 팀원·에이전트가 로그 위치를 한 곳에서 찾을 수 있다.

**영향**

- CLAUDE.md·AGENTS.md §3·§11·§11.5, README.md의 경로 참조를 일괄 갱신했다.
- `.gitignore`의 `logs/` 제외 패턴(런타임 로그용)이 팀 로그 폴더를 통째로 제외하게 되므로 해당 패턴을 제거했다(런타임 로그는 `*.log`와 `out/`으로 계속 제외).
- 모든 팀원 에이전트는 이후 `logs/` 하위 경로에 기록해야 한다.

**재검토 조건**

- 런타임 로그 폴더가 별도로 필요해져 이름 충돌이 생기는 경우.
