# 회의록 생성 (Notion 녹취록 → 구조화 요약)

이 스킬은 CareerPath AX 팀 전용이다. Notion 녹음으로 생성된 녹취록을 **읽어서**
회의별 **구조화 요약본** 한 개를 만든다. 규칙은 `CLAUDE.md §11.6`을 따르고,
요약 형식은 같은 폴더의 `summary-guide.md`를 따른다.

## 호출 방법
- 슬래시 명령: **`/meeting-notes`**
- 또는 그냥 말로: **"회의록 만들어줘"**, "녹취록 정리해줘", "회의 요약해줘" 등
- (선택) Notion 회의 페이지 URL을 함께 주면 검색을 건너뛴다: "이 회의 회의록 만들어줘 &lt;URL&gt;"

> **원문(받아쓰기 전문)은 git에 저장하지 않는다.** 녹취록 전사에는 개인정보·사담이 섞일 수
> 있어, 원문은 접근 권한이 통제되는 팀 Notion에만 두고 재사용한다. 스킬은 원문을 요약의
> 근거로만 읽고, 로컬 파일로는 만들지 않는다.

## 설정 (첫 실행 전에 팀이 채운다)

```yaml
NOTION_회의록_DB_URL: https://www.notion.so/810f1a683b4f8256b185016b28b0514b?v=643f1a683b4f829483fc08e1cb96a406
NOTION_회의록_DATA_SOURCE: collection://3c8f1a68-3b4f-8257-8092-07aa8ed625fa   # SQL 쿼리용
NOTION_회의록_기본_VIEW: https://www.notion.so/810f1a683b4f8256b185016b28b0514b?v=643f1a683b4f829483fc08e1cb96a406   # "모든 회의"(날짜 내림차순)
```

- 이 DB는 팀이 만든 커스텀 **"회의록 DB"** 다(Notion 기본 meeting-notes 소스가 아님). 각 행(페이지)이 한 회의이며, 페이지 본문에 Notion 녹음 녹취록이 들어 있다.
- 스키마: `회의 이름`(제목) · `날짜`(date) · `카테고리`(정규 회의/짧은 회의) · `참석자`(person) · `요약`(text) · `생성자`.
- 값이 비어 있으면(`<채워주세요>`) **워크스페이스 전체 검색(`notion-search`)으로 폴백**하고, 사용자에게 "DB URL을 채우면 더 정확합니다"라고 한 번 안내한다.

## 사전 조건

- **Notion 커넥터/MCP 인증이 필요하다.** 인증돼 있지 않으면 Notion 조회가 실패한다.
  이때는 사용자에게 (1) Notion 커넥터를 인증해 달라고 안내하거나, (2) 폴백으로 녹취록
  텍스트를 직접 붙여넣어 달라고 요청한다(붙여넣기 폴백이면 3단계를 건너뛰고 4단계부터 진행).

## 실행 절차

### 1. 입력 확인
- 사용자가 Notion 페이지 URL을 함께 줬으면 그 페이지를 대상으로 하고 2·3단계를 건너뛴다.
- URL이 없으면 검색으로 진행한다.

### 2. Notion 인증 확인
- Notion MCP(`notion-fetch` 등)가 사용 가능한지 확인한다. 불가하면 위 '사전 조건'대로 안내/폴백한다.

### 3. 회의 검색 → 사용자 선택
- 기본: `notion-query-data-sources`를 **view 모드**로 호출해 기본 뷰("모든 회의", 날짜 내림차순) 목록을 받는다.
  ```
  notion-query-data-sources { data: { mode: "view", view_url: "<NOTION_회의록_기본_VIEW>", page_size: 10 } }
  ```
  (특정 회의명으로 좁히려면 SQL 모드로 `NOTION_회의록_DATA_SOURCE`에 `WHERE "회의 이름" LIKE ?` 사용.)
- DB URL 미설정이면 `notion-search`로 워크스페이스에서 회의/녹취록 페이지를 조회하는 폴백.
- **후보를 제시**한다: `회의 이름 · 날짜 · 카테고리 · URL`. 사용자가 하나를 고르게 한다.
  (여러 개를 한 번에 처리하라고 하면 각 항목을 순차 처리.)

### 4. 녹취록 원문 읽기 (저장하지 않음)
- 선택 페이지를 **녹취록 포함**으로 가져온다:
  ```
  notion-fetch { id: "<선택한 회의 페이지 URL>", include_transcript: true }
  ```
  (`include_transcript: true`가 없으면 녹취록 대신 자리표시자만 나온다. 반드시 true로.)
- 이 원문은 **요약의 근거로만 사용**한다. **받아쓰기 파일로 저장하지 않는다.** 원문은 Notion에 그대로 둔다.
- fetch 결과에서 `createdTime`(Notion 생성일)과 `회의 이름`을 확보한다.

### 5. 저장 파일 결정 (파일명 = 날짜_회의명)
- 파일 경로: `logs/meetinglogs/YYYY-MM-DD_<회의명-slug>.md` (폴더 없이 단일 파일).
- 날짜는 **`createdTime`(Notion 생성일)** 의 `YYYY-MM-DD`. 사용자 로컬(KST) 기준으로 판단한다. **오늘 날짜가 아니다.** (참고: 생성일과 회의 `날짜` 속성이 다르면 기본은 생성일을 쓰되, 팀이 원하면 `날짜` 속성 기준으로 바꿀 수 있다.)
- `<회의명-slug>`: 회의 이름의 공백을 `-`로, 경로에 못 쓰는 특수문자는 제거.
- 같은 이름의 파일이 이미 있으면 뒤에 `_2` 등을 붙인다.

### 6. 구조화 요약 저장 → `YYYY-MM-DD_<회의명>.md`
- 위 경로에 요약 파일을 만든다. **형식·규칙은 반드시 `summary-guide.md`를 따른다.**
- 원문 참조는 로컬 파일이 아니라 **Notion 페이지 URL**로 건다(원문은 Notion에만 있으므로).
- **개인 신상·사담 등 회의 외적 민감정보는 요약에 옮기지 않는다**(summary-guide.md의 개인정보 규칙 준수).

### 7. 결정·액션 보고 (등록은 사람이)
- 요약을 끝낸 뒤, 요약 중 발견한 **팀 결정 후보**와 **액션 아이템**을 사용자에게 **보고**한다.
- **자동으로 다른 파일을 고치지 않는다.** 다음을 안내만 한다:
  - 팀 전체 결정이면 → 사람이 검토 후 `logs/decisionlogs/_team.md`에 `D-NNN`으로 등록
  - 액션 아이템(할 일)은 → **이 회의록 안에서만 관리**한다. worklog에 넣지 않는다(worklog는 완료 이력). 담당자가 그 일을 완료하면 그때 자기 worklog에 완료로 남긴다.

### 8. 완료 보고 + worklog 안내
- CLAUDE.md §12 완료 보고 형식으로 보고한다(작업자·변경 파일·검증·발견된 결정/액션 포함).
- 회의록을 가져온 사람은 자기 개인 worklog에 `W-<코드>-NNN`으로 이 작업을 남긴다.

## 주의
- **원문 전사는 로컬/ git에 저장하지 않는다.** 원문은 팀 Notion에만 둔다. 한 번 원격에 올라간 개인정보는 히스토리 재작성으로도 완전히 되돌리기 어렵기 때문이다.
- 개인정보·민감정보(급여·연락처·개인 신상·사담 등)는 요약에도 넣지 않는다. 꼭 필요한 회의 맥락만 남긴다. 판단이 애매하면 사용자에게 확인한다(CLAUDE.md §7).
- 이 스킬 파일은 팀 공용이다. 규칙 변경은 팀에 공유한다(단, `CLAUDE.md`/`AGENTS.md` 수정은 김재천만).
