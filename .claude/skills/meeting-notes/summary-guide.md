# 구조화 요약 지침 (`요약.md` 작성 규칙)

`meeting-notes` 스킬이 `요약.md`를 만들 때 따르는 형식과 규칙이다. 목적은 **바쁜 팀원이 3분 안에
회의 내용을 파악**하고, 회의에서 나온 **결정·할 일을 놓치지 않게** 하는 것이다.

## 작성 원칙 (CLAUDE.md 준수)
- **초급자도 이해하게**: 기술 용어는 처음 나올 때 한 줄로 설명한다.
- **불확실은 `확인 필요`** 로 표기하고 단정하지 않는다.
- **표현 규칙(PRD 준수)**: "합격 가능성", "취업 성공 확률" 같은 표현은 쓰지 않는다. 점수는
  "공고 적합도 점수" 등으로, 데이터 근거는 "샘플 데이터 기준"으로 적는다.
- **원문 왜곡 금지**: 녹취록에 없는 내용을 지어내지 않는다. 애매하면 원문 표현을 인용한다.
- **개인정보·사담 제외(중요)**: 원문 전사는 git에 저장하지 않고 Notion에만 둔다. 요약에도
  개인 신상·급여·연락처·사적 잡담 등 회의 외적 민감정보는 옮기지 않는다. 회의 진행에 필요한
  맥락만 남기고, 판단이 애매하면 사용자에게 확인한다(CLAUDE.md §7). 이 요약 파일은 git으로
  공유되므로, 한 번 올라가면 되돌리기 어렵다는 점을 전제로 작성한다.
- **결정·액션의 위치**: 요약은 결정·액션을 *발견해 표시*할 뿐, 다른 파일을 직접 고치지 않는다.
  - **팀 결정**은 사람이 검토 후 `_team.md`(또는 개인 decisionlog)에 등록한다.
  - **액션 아이템(할 일)은 이 회의록 안에서만 관리**한다. worklog에는 넣지 않는다 — worklog는 *완료 이력*이기 때문이다. 담당자가 그 일을 **실제로 완료하면 그때** 자기 worklog에 완료 작업으로 남긴다.

## 요약 파일 표준 형식 (`YYYY-MM-DD_<회의명>.md`)

```markdown
# 회의록 · <회의명>

- **일시(Notion 생성일시)**: <created_time>
- **참석**: <이름들> (불참: <이름>)
- **기록자(가져온 사람)**: <이름(코드)>
- **원문(녹취록 전사)**: 팀 Notion에서만 보관·공유 → <Notion 페이지 URL>

## TL;DR (한 줄 요약)
- <회의의 핵심을 한두 문장으로>

## 안건별 논의 요약
### 1. <안건 제목>
- <논의된 핵심 내용>
- <이견이 있었다면 양쪽 입장>

### 2. <안건 제목>
- ...

## 결정 사항 (후보 — 등록은 사람이)
- [ ] <결정 내용> — 구분: **팀 전체 결정** → `logs/decisionlogs/_team.md`에 `D-NNN` 등록 대상
- [ ] <결정 내용> — 구분: **개인 결정(누구)** → 해당 팀원 `decisionlogs/<이름>.md`(`D-<코드>-NNN`) 대상
- (결정이 없으면 "이번 회의에서 확정된 결정 없음"이라고 적는다)

## 액션 아이템 (이 회의록에서 관리 — worklog에 옮기지 않음)
| 담당 | 할 일 | 기한 | 상태 |
|------|-------|------|------|
|      |       |      | 미진행 |
> 담당자가 완료하면, 완료 사실은 그때 담당자 worklog에 기록한다(여기 액션은 할 일 목록일 뿐).

## 프로젝트 영향
회의 내용이 아래에 영향을 주면 짚어준다(없으면 "해당 없음"):
- **일정/제출**: 참가 신청(~2026-07-10 13:00), 예선 과제 제출(~2026-08-10 13:00), 본선(9월 초)에 영향?
- **MVP 범위**: 반드시 구현/제외 기능이 바뀌었나? (PRD §10 참고)
- **데이터·기술 스택**: 채용공고/훈련과정 데이터, TF-IDF·LLM·Streamlit 등 관련 결정?
- **역할 분담**: 6인(서윤범 팀장·김윤서·김재천·김하연·박하린·함다연) 중 누가 무엇을 맡기로?
- **리스크**: 새로 드러난 위험과 대응(API 접근, 개인정보, 데이터 확보 등)

## 미결 / 다음 회의 안건
- <이번에 결론 못 낸 것>
- 다음 회의 일시·안건: <있으면>
```

## 팀 결정 vs 개인 결정 판단 기준
- **팀 전체 결정**(→ `_team.md`): 문제 선정, 기술 스택, 구현 범위 확정·축소, 데이터 처리,
  보안·개인정보 처리, 검증 방식, 배포·제출·공유 방식, 큰 구조 변경, 역할 분담.
- **개인 결정**(→ `decisionlogs/<이름>.md`): 특정 팀원이 자기 담당 작업에서 내린 방향 결정.
- 판단이 애매하면 팀 전체 결정 후보로 올려 사람이 정하게 한다.

## 스킬이 요약 후 보고할 것
요약을 저장한 뒤, 사용자에게 다음을 요약해 보고한다(자동 등록 금지):
1. 발견한 **팀 결정 후보** 목록 (→ 사람이 `_team.md` 등록 권유)
2. 정리한 **액션 아이템** 목록 (→ 이 회의록에서 관리. worklog로 옮기지 않음)
3. `확인 필요`로 남긴 불확실 항목
