# CLAUDE.md · CareerPath AX for Counselors 작업 헌법

이 문서는 이 프로젝트에서 AI 협업 에이전트가 **매번 지시하지 않아도 항상 따라야 하는 기준**이다. 팀원과 에이전트 모두 이 규칙을 기준으로 일한다. 규칙이 바뀌면 이 문서를 먼저 갱신하고 팀 공용 결정 로그(`logs/decisionlogs/_team.md`)에 이유를 남긴다.

---

## 🔒 수정 권한 (OWNERSHIP LOCK)

> 이 파일(`CLAUDE.md`, `AGENTS.md`)은 팀원 **김재천(KJC)** 과 김재천의 에이전트만 수정·삭제할 수 있다. 다른 팀원(서윤범·김윤서·김하연·박하린·함다연) 및 그들의 에이전트는 이 두 파일을 **수정·삭제할 수 없다.**
>
> - 에이전트는 이 두 파일에 대한 편집/삭제 요청을 받으면, 현재 작업자가 **김재천임이 확인되지 않는 한 작업을 거부**하고 다음과 같이 안내한다: "CLAUDE.md와 AGENTS.md는 김재천만 수정할 수 있습니다. 변경이 필요하면 김재천에게 요청하세요."
> - 규칙 변경 제안은 `logs/decisionlogs/_team.md` 또는 팀 채널을 통해 김재천에게 전달한다.
> - 이 잠금은 두 파일의 **모든 편집·부분 수정·삭제**에 적용된다. 오탈자 수정도 예외가 아니다.

## 🔁 동기화 규칙 (SYNC)

> `CLAUDE.md`와 `AGENTS.md`는 항상 **1:1로 동일한 운영 규칙**을 유지한다. **`CLAUDE.md`가 기준(SSOT, Single Source of Truth)** 이고, `AGENTS.md`는 이를 그대로 반영한다.
>
> - 한쪽을 수정하면 **같은 변경 안에서** 다른 쪽도 동일하게 수정한다.
> - 두 파일의 규칙 내용이 다르면 그 자체가 오류이며, 김재천이 즉시 일치시킨다.
> - 두 파일 모두 위 OWNERSHIP LOCK의 적용을 받는다(김재천만 수정).

## 1. 대화와 보고 방식
- 결론을 먼저, 근거·세부는 뒤에.
- 기술 용어는 처음 등장할 때 한 줄로 설명한다.
- 불확실하면 단정하지 말고 `확인 필요`로 표기한다.
- 실행/확인하지 않은 것을 완료라고 말하지 않는다.
- 모든 응답은 격식 있는 한국어 대화체(합쇼체/하십시오체)로 작성한다.

## 2. 작업 시작 전 확인
- 현재 폴더 구조와 기존 파일을 먼저 확인한다.
- 기존 파일·다른 팀원의 변경사항을 함부로 덮어쓰지 않는다.
- 바로 구현하지 말고, 영향 범위와 계획을 먼저 정리한다.
- 팀 작업이므로, 다른 팀원이 담당하는 영역을 수정하기 전에는 해당 팀원의 로그·결정을 먼저 확인한다.

## 3. 파일 구조 원칙
- `src/` 실행 코드 · `tests/` 테스트 · `tools/` 보조 스크립트 · `docs/` 문서 · `out/` 실행·임시 산출물(Git 제외) · `data/raw/` 원본 · `data/processed/` 가공.
- 사용자가 직접 실행하는 핵심 기능은 `src/`. 개발자 전용 보조 스크립트는 `tools/`. 반복 실행 결과는 `out/`.
- 운영 로그는 `logs/` 하위로 통합: `logs/worklogs/`(개인별), `logs/decisionlogs/`(개인별 + 공용 `_team.md`), `logs/Troubleshootinglog.md`(공용 단일).
- 필요 없는 폴더는 만들지 않고 이유를 남긴다.

## 4. 구현 원칙
- 처음부터 전체를 달리지 않고 가장 작은 성공 단위부터 만든다.
- 각 단계는 다음을 명확히 한다: 왜 필요한가 / 이전 단계와 연결 / 다음 단계로 전달 / 완료 조건 / 검증 방법 / 실패 시 중단점.
- 단계 완료 보고에 포함: 전체 중 현재 위치, 이번에 한 일, 변경 파일, 검증 결과, 남은 단계, 예상 잔여량, 발생한 문제·T-ID, 필요 시 Git commit/push 여부.

## 5. 기준 계약 문서 원칙
- 구현 전 `docs/requirements-contract.md`를 작성한다: 입력 형식 / 출력 형식 / 주요 지표 정의 / 수식 / 상태 판정 / 오류 판정 / 허용·비허용 값 / 누락 데이터 처리 / 시간대·단위·반올림 / 완료 기준.
- 같은 지표가 파일마다 다르게 정의되지 않게 한다. 기준이 바뀌면 코드보다 계약 문서를 먼저 갱신하고 이유를 기록한다.

## 6. 테스트와 검증 원칙
- 영향 범위에 맞춰 검증한다: 단위·통합·입력·오류 상황·경계값 검증, 문서-코드 정합성, README-기능 일치, 제출물/배포물 구조 확인.
- 검증하지 못한 부분은 숨기지 말고 `미검증 범위`로 보고한다. 실패한 테스트를 삭제·완화해 통과시키지 않고 원인을 수정한다.

## 7. 보안·개인정보·비밀정보 원칙
- 다음을 코드·문서·로그·테스트 데이터에 넣지 않는다: API 키, 비밀번호, 액세스 토큰, 세션 쿠키, 실제 고객정보, 실제 계좌정보, 민감한 내부 정보.
- 구직자 개인정보(이력서·자기소개서·포트폴리오 원문)는 MVP에서 저장하지 않는다. 데모는 가상 페르소나·익명화 샘플만 사용한다.
- 원본 데이터는 수정하지 않는다. 필요하면 마스킹/정규화된 파생 파일을 별도로 만든다.
- 비밀정보가 발견되면 임의로 로그를 편집하지 말고, 노출 범위와 안전한 교체·폐기 방법을 사용자에게 보고한다.

## 8. 문서화 원칙

- 기획서(PRD)·기준 계약 문서·코드·테스트·README·제출/배포 설명·worklog·decisionlog가 서로 모순되지 않게 관리한다.
- 리포트·발표 문구는 PRD의 표현 원칙을 따른다(예: "합격 가능성" 금지, "공고 적합도 점수" 사용, "샘플 데이터 기준" 라벨).

## 9. 외부 자료 사용 원칙

- 외부 출처는 필요 시 `docs/references/`에 정리한다.

## 10. Git 원칙
- 주요 단계가 끝날 때마다: 변경 파일 확인 → 불필요한 파일 제외 확인 → 테스트/검증 실행 → commit → push.
- 로그성 산출물, 비밀정보, 로컬 실행 결과, 임시 산출물(`out/`), 원본 데이터(`data/`)는 기본적으로 Git에 올리지 않는다.
- 팀 작업이므로 커밋 메시지에 어떤 작업(W-ID)·결정(D-ID)과 연결되는지 남기면 추적이 쉽다.

## 11. 로그 기록 규칙

로그는 **개인별 폴더 + 공용 단일 문서** 구조로 관리한다.

### 11.1 Worklog — `logs/worklogs/<이름>.md` (개인별, 주요 요청마다)
- 각 팀원은 자신의 파일(`logs/worklogs/서윤범.md` 등)에 기록한다. 다른 팀원의 worklog 파일은 수정하지 않는다.
- 모든 주요 사용자 요청이 끝날 때마다 기록한다. 설명 작업이라도 방향·구조·판단에 영향을 주면 기록한다. 최신 항목을 위에 추가한다.
- ID: `W-<코드>-NNN` (예: 김재천 → `W-KJC-001`). 팀 전체에서 고유하도록 코드를 접두한다.
```
### W-KJC-001 · 작업 제목
**요청** / **수행 작업** / **변경 파일** / **검증**(미실행 시 사유) / **판단 근거** / **결과**(완료·남은 작업)
```

### 11.2 Decisionlog(개인) — `logs/decisionlogs/<이름>.md` (방향을 바꾸는 결정만)
- 각 팀원이 개인 작업에서 내린 중요한 결정을 자신의 파일에 기록한다. 사소한 수정은 worklog만.
- ID: `D-<코드>-NNN` (예: `D-SYB-001`).
```
### D-SYB-001 · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### 11.3 Decisionlog(팀 공용) — `logs/decisionlogs/_team.md`
- 팀 전체 방향 결정만 기록한다. 대상: 문제 선정, 기술 스택 선택, 구현 범위 확정·축소, 데이터 처리 방식, 보안·개인정보 처리, 테스트·검증 방식, 배포·제출·공유 방식, 큰 구조 변경.
- ID: `D-NNN` (코드 없음). **제품 방향 결정 D-001~D-013은 `docs/prd/careerpath-ax-planning.md §14`에 있으므로, 신규 팀 결정은 `D-014`부터 이어서 번호를 붙인다.**
```
### D-014 · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### 11.4 Troubleshootinglog — `logs/Troubleshootinglog.md` (공용 단일, 전 팀원 기록)
- 실제 오류·실패·환경 문제·검증 실패·설계 충돌이 발생하면 이 **하나의 문서**에 기록한다. 누구의 기록인지 **담당자** 필드로 명시한다.
- 다른 팀원의 로컬 에이전트도 이 문서에서 해결책을 재사용할 수 있도록, **재사용 가능한 해결책**과 **후속 조치/대응 방안**을 반드시 채운다.
- ID: `T-NNN` (팀 전체 순차, 코드 없음).
```
### T-001 · 문제 제목
- **담당자(기록자)**: 이름 (에이전트/사람)
- **발생 상황** / **증상** / **확인된 원인**(+불확실점) / **조치 / 최종 해결**
- **재사용 가능한 해결책** / **후속 조치 / 대응 방안** / **상태**(미해결·해결됨·재발 관찰 중) / **재발 방지** / **재발 이력**
```

## 11.5 문제 대응 규칙 (T-코드 우선 재사용)
작업 중 오류·실패·환경 문제가 발생하면 다음 순서를 따른다.
1. 새 T-ID를 만들기 전에, `logs/Troubleshootinglog.md`에서 **동일·유사 증상의 T-코드가 있는지 먼저 검색**한다.
2. 있으면 해당 항목의 **'재사용 가능한 해결책'을 우선 적용**한다.
3. 적용해서 해결되면 그 T-ID의 **'재발 이력'** 에 (작업자·날짜·적용 결과)를 덧붙인다.
4. 기존 해결책으로 해결되지 않거나 새로운 문제면 **새 T-ID를 생성**하고, 다른 팀원 에이전트가 재사용할 수 있도록 '재사용 가능한 해결책'과 '후속 조치/대응 방안'을 반드시 채운다.

## 12. 완료 보고 기준
작업 완료 시 다음 형식으로 보고한다. 실행하지 않은 검증은 완료라고 하지 않는다.
```
완료했습니다.
- 작업자(팀원):
- 수행한 작업:
- 변경 파일:
- 검증 결과:
- 기록: Worklog(W-ID) / Decisionlog(D-ID) / Troubleshootinglog(T-ID):
- 남은 작업:
- 한계 또는 미검증 범위:
- Git 상태:
```
